PART 01

Fundamentals

Core Intuition — Why Do Linked Lists Exist?

Arrays are stored in contiguous memory. This is their superpower (O(1) random access via index) and their curse (insertion/deletion in the middle requires shifting every subsequent element — O(n)).

Linked lists trade random access for flexible insertion and deletion. Each node lives wherever in memory, connected to the next by a pointer. To insert between two nodes, you just rewire two pointers — O(1). No shifting. No reallocation.

Mental Model
Think of an array as a row of numbered seats in a cinema — fixed positions, you can jump to seat 42 instantly, but inserting a new seat forces everyone to shuffle. A linked list is a treasure hunt — each clue (node) tells you where the next clue is. You can insert a new clue anywhere by just changing two references, but to get to clue 42, you must follow all 41 before it.

Trade-off Summary

Operation Array Linked List
Access by index O(1) O(n)
Search O(n) O(n)
Insert at beginning O(n) — shift all O(1)
Insert at end O(1) amortized O(1) with tail pointer, else O(n)
Insert in middle O(n) O(1) given a pointer to that node
Delete O(n) O(1) given pointer to prev node
Memory Compact — values only Extra pointer(s) per node
Cache friendliness High — contiguous memory Low — nodes scattered in heap
Key Mental Shift
In arrays, you think about indices. In linked lists, you think about pointers. The index is replaced by a reference to the next node. "Moving forward" means following a pointer, not incrementing an index. This is the mental model you must internalize before solving any linked list problem.

Types of Linked Lists

1. Singly Linked List

Each node holds a value and a single pointer to the next node. The last node points to null.

[1] ──next──▶ [2] ──next──▶ [3] ──next──▶ [4] ──next──▶ null head tail

When to use: The default choice. Simple queues, stacks, most interview problems. Use when you only need forward traversal.

  • Forward traversal only.
  • To delete a node, you need a reference to its previous node.
  • Most LeetCode linked list problems use singly linked lists.

2. Doubly Linked List

Each node holds a value, a pointer to the next node, and a pointer to the previous node.

null ◀──prev── [1] ⟺ [2] ⟺ [3] ⟺ [4] ──next──▶ null Each node: prev ◀── [val] ──▶ next

When to use: When you need bidirectional traversal. LRU Cache (delete a node in O(1) without searching for its predecessor). Browser history (back and forward). Text editors (cursor movement). Java's LinkedList class is doubly linked internally.

  • Can traverse forward and backward.
  • Can delete any node in O(1) given only a pointer to that node (use node.prev).
  • Uses 2x the pointer memory compared to singly.

3. Circular Linked List

The last node's next pointer points back to the head (or any earlier node) instead of null. Can be singly or doubly circular.

┌─────────────────────────────────────────┐ ▼ │ [1] ──▶ [2] ──▶ [3] ──▶ [4] ──next──┘

When to use: Round-robin scheduling (processes taking turns), music playlists that loop, multiplayer games (each player's turn cycles). Detecting cycles with Floyd's algorithm is fundamentally about recognizing circular behavior.

  • No natural null terminator — traversal must stop based on a condition (e.g., returning to head).
  • Can start traversal from any node.
  • Infinite loop risk if you forget the stopping condition.

Difference Summary

Property Singly Doubly Circular
Pointers per node 1 (next) 2 (prev + next) 1 or 2 (wraps)
Traversal direction Forward only Both directions Forward, loops
Delete given only node ref Need prev node O(1) — use node.prev Depends on type
Tail points to null null head (or any node)
Cycle detection risk No (if correct) No (if correct) Always "cyclic"
Real-world use Stacks, queues, most DSA LRU Cache, browsers Schedulers, games
PART 02

Pointer Techniques — The Heart of Linked Lists

This is the most critical section. Linked list problems are 95% pointer manipulation. You are not processing values — you are rewiring connections between nodes. Master this, and almost every linked list problem becomes straightforward.

Pattern 1 — Traversal

The foundation. Moving from head to tail by following next pointers.

ListNode curr = head;
while (curr != null) {
    // process curr.val here
    curr = curr.next; // advance: follow the pointer
}
// After loop: curr == null, you have reached the end
Critical Mistake
while (curr.next != null) stops ONE node early (at the last node). while (curr != null) processes every node including the last. Know which you need. For finding the tail node, you want while (curr.next != null). For processing all values, use while (curr != null).

Pattern 2 — Reversal (The Most Asked Pattern)

Reversing a linked list requires tracking three pointers simultaneously: prev, curr, and next. The operation is: for each node, make it point backward instead of forward.

Mental Model — "Breaking and Reconnecting Links"
Imagine each arrow between nodes is a physical wire. To reverse, you snip the wire pointing forward and reconnect it to point backward. But before snipping, you must save where the forward wire was going — otherwise you lose the rest of the list. This "save before modify" discipline is the entire reversal pattern.
// Dry Run — Reversing: 1 → 2 → 3 → 4 → null
0
Initial state: prev = null, curr = [1]. The list is 1→2→3→4→null. We want null←1←2←3←4.
1
Iteration 1: Save next = curr.next = [2]. Set curr.next = prev = null (node 1 now points backward to null). Advance: prev = curr = [1], curr = next = [2]. State: null ← [1], curr=[2].
2
Iteration 2: Save next = curr.next = [3]. Set curr.next = prev = [1] (node 2 now points to node 1). Advance: prev = [2], curr = [3]. State: null ← [1] ← [2], curr=[3].
3
Iteration 3: Save next = [4]. Set [3].next = [2]. Advance: prev = [3], curr = [4]. State: null ← [1] ← [2] ← [3], curr=[4].
4
Iteration 4: Save next = null. Set [4].next = [3]. Advance: prev = [4], curr = null. Loop ends. Return prev = [4] as the new head. Result: null ← 1 ← 2 ← 3 ← 4, read as 4→3→2→1→null. ✓

Pattern 3 — Fast & Slow Pointers (Floyd's Algorithm)

Two pointers start at head. slow moves one step at a time. fast moves two steps at a time. This pattern exploits the speed differential to detect cycles, find the middle, and locate the Nth node from the end.

Mental Model — "The Track Runner"
Imagine two runners on a circular track. The faster one laps the slower one — they meet. On a straight track, the fast runner reaches the end when the slow runner is at the midpoint. This is the core intuition behind every fast & slow pointer problem. 
ListNode slow = head;
ListNode fast = head;

while (fast != null && fast.next != null) {
    slow = slow.next;       // 1 step
    fast = fast.next.next;  // 2 steps
}
// For a list of length n:
// - No cycle: fast reaches null, slow is at the middle node
// - Cycle: fast and slow will eventually meet inside the cycle
Critical: Always check fast AND fast.next
The condition must be fast != null && fast.next != null. If you only check fast != null, then fast.next.next will throw a NullPointerException when fast is at the last node. Both checks are mandatory.

Pattern 4 — Dummy Node Technique

Create a fake "dummy" node before the real head. Point its next to the actual head. Operate on the list. Return dummy.next as the new head.

Why? Without a dummy node, inserting before the head, or deleting the head node, requires special-case logic. With a dummy node, the head is just another node — no special cases. This simplifies code dramatically. 

ListNode dummy = new ListNode(0); // dummy.val doesn't matter
dummy.next = head;               // link dummy before the real list

ListNode curr = dummy;           // operate using curr
// ... perform operations ...

return dummy.next; // new head (may differ from original head)
When to Always Use Dummy Node
Merge two sorted lists, remove Nth from end, delete a node by value, partition list, add two numbers. Essentially: whenever the head might change, or when you need a "previous" pointer at the very start of the list.

Mental Models — "Tracking Previous, Current, Next"

Before ANY pointer reassignment in a linked list, ask yourself three questions:

  1. What am I about to overwrite? — Save it in a temp variable first.
  2. What should this pointer now point to? — Make the assignment.
  3. How do I advance to the next node? — Use the saved reference.

This "save → reassign → advance" discipline eliminates 90% of linked list bugs. Never reassign a pointer until you've saved where it was going.

Common Mistakes — What Breaks Logic

!
Losing reference to next before saving. The most common bug. If you do curr.next = prev before saving next = curr.next, you have permanently lost the rest of the list. The order is always: save first, then modify.
!
Null pointer errors. Accessing curr.next when curr is null. Accessing fast.next.next when fast.next is null. Always guard pointer dereferences with null checks.
!
Incorrect head update. After reversal or structural changes, failing to return the correct new head. In reversal, the new head is prev (not curr, which is null). Always trace through where head ends up.
!
Creating an unintended cycle. When linking the tail of one sublist to the head of another, if you forget to set the tail's next to null, you may accidentally create a cycle. Always null-terminate modified sublists.
!
Off-by-one in N-from-end problems. If you move fast N steps ahead and then move both until fast reaches null vs. the last node — these differ by 1. Dry run edge cases with small lists (n=1, n=length).
PART 03

Core Operations — Intuition + Dry Run

Insert — At Beginning (O(1))

Create a new node. Point its next to the current head. Update head to the new node.

void insertAtBeginning(int val) {
    ListNode newNode = new ListNode(val);
    newNode.next = head; // new node points to old head
    head = newNode;      // update head to new node
}
// Order matters: if you set head = newNode FIRST,
// you lose the reference to the old head — newNode.next becomes null.
// Insert 0 at beginning of: 1 → 2 → 3
1
Create newNode = [0]. List: [0], head=[1]→[2]→[3].
2
Set newNode.next = head. Now [0]→[1]→[2]→[3]. head still = [1].
3
Set head = newNode. Now head=[0]→[1]→[2]→[3]. ✓

Insert — At End (O(n) without tail pointer)

Traverse to the last node (where curr.next == null). Link the new node after it.

void insertAtEnd(int val) {
    ListNode newNode = new ListNode(val);
    if (head == null) { head = newNode; return; } // empty list edge case

    ListNode curr = head;
    while (curr.next != null) { // stop at last node (not null!)
        curr = curr.next;
    }
    curr.next = newNode; // link new node after last
}

Insert — At Middle (Given Position)

Traverse to the node before the target position. Rewire two pointers.

void insertAtPosition(int pos, int val) {
    if (pos == 0) { insertAtBeginning(val); return; }

    ListNode newNode = new ListNode(val);
    ListNode curr = head;

    for (int i = 0; i < pos - 1 && curr != null; i++) {
        curr = curr.next; // stop at node BEFORE insertion point
    }
    if (curr == null) return; // position out of range

    newNode.next = curr.next; // step 1: new node points to successor
    curr.next = newNode;      // step 2: predecessor points to new node
    // NEVER swap these two lines — you'd lose the successor reference
}
Order Is Everything
In middle insertion: always set newNode.next = curr.next BEFORE curr.next = newNode. If you do it backward, curr.next (the successor) is overwritten first, and you lose the rest of the list.

Delete — By Value

Track the previous node. When you find the node with target value, skip it by linking previous directly to the node after it.

ListNode deleteByValue(ListNode head, int val) {
    ListNode dummy = new ListNode(0);
    dummy.next = head;
    ListNode prev = dummy;

    while (prev.next != null) {
        if (prev.next.val == val) {
            prev.next = prev.next.next; // skip the target node
            // do NOT advance prev — there may be another duplicate
        } else {
            prev = prev.next; // only advance if no deletion
        }
    }
    return dummy.next;
}
// Delete value 3 from: 1 → 3 → 3 → 4
1
dummy→[1]→[3]→[3]→[4]. prev=dummy. prev.next=[1], val≠3, advance. prev=[1].
2
prev.next=[3], val==3. Set prev.next = [3].next = [3] (second 3). Do NOT advance prev. prev=[1].
3
prev.next=[3] (second), val==3. Set prev.next = [3].next = [4]. Do NOT advance.
4
prev.next=[4], val≠3, advance. prev=[4]. Loop ends. Return dummy.next = [1]→[4]. ✓

Delete — By Position (Remove Nth from End)

Use the fast & slow pointer gap trick: advance fast N steps ahead, then move both until fast.next == null. slow is now at the node before the target.

ListNode removeNthFromEnd(ListNode head, int n) {
    ListNode dummy = new ListNode(0);
    dummy.next = head;
    ListNode fast = dummy, slow = dummy;

    for (int i = 0; i <= n; i++) { // advance fast by n+1 steps
        fast = fast.next;
    }
    while (fast != null) {  // move both until fast reaches end
        slow = slow.next;
        fast = fast.next;
    }
    slow.next = slow.next.next; // skip the Nth-from-end node
    return dummy.next;
}

Search

Linear traversal — O(n). No shortcuts in an unsorted list.

ListNode search(ListNode head, int target) {
    ListNode curr = head;
    while (curr != null) {
        if (curr.val == target) return curr; // found
        curr = curr.next;
    }
    return null; // not found
}

Reverse — Iterative

Three pointers: prev, curr, next. For each node: save next, flip the link, advance. Time: O(n). Space: O(1).

ListNode reverseIterative(ListNode head) {
    ListNode prev = null;
    ListNode curr = head;

    while (curr != null) {
        ListNode next = curr.next; // ① SAVE: preserve reference forward
        curr.next = prev;          // ② FLIP: point current node backward
        prev = curr;               // ③ ADVANCE prev
        curr = next;               // ④ ADVANCE curr using saved reference
    }
    return prev; // prev is the new head (curr is null)
}
// Common mistake: returning curr (which is null) instead of prev.

Reverse — Recursive

Recurse to the end of the list. As the call stack unwinds, reverse each link. The new head is the last node found during the recursion.

ListNode reverseRecursive(ListNode head) {
    // Base: empty or single node — already reversed
    if (head == null || head.next == null) return head;

    // Recurse: reverse the rest of the list first
    ListNode newHead = reverseRecursive(head.next);

    // Now head.next still points forward. Reverse it:
    head.next.next = head; // the next node now points back to head
    head.next = null;     // sever head's forward link

    return newHead; // bubble the new head all the way up
}
// Space: O(n) call stack. Use iterative for large lists.
// Key insight: head.next.next = head flips the last remaining link at each frame.
Iterative vs Recursive Reversal
Both are O(n) time. Iterative is O(1) space. Recursive uses O(n) stack space. In interviews, know both — but prefer iterative unless asked for recursive. The recursive version is elegant for understanding, but dangerous on very long lists (stack overflow).
PART 04

Advanced Patterns

Every linked list interview problem maps to one (or a combination) of these seven patterns. Recognize the pattern, apply the template, adapt.

A · Full & Partial Reversal

Reverse the entire list or a subrange [left, right]. Requires careful boundary handling and often a dummy node.

B · Fast & Slow Pointer

Speed differential to find cycles, midpoints, and Nth-from-end nodes. The most versatile linked list technique.

C · Merge / Two-Pointer

Merge two sorted lists, merge K lists, or use two pointers moving in tandem across two separate lists.

D · Palindrome Check

Find middle, reverse second half, compare. Classic compound problem combining multiple patterns.

E · Reorder / Restructure

Interleave, zip, or reorder nodes without creating new nodes. Pointer rearrangement under constraints.

F · Intersection

Find where two lists merge. Use length difference or the two-pointer cycling trick to align starts.

G · K-way Operations

Reverse in K-groups, merge K sorted lists. Generalizes reversal and merge with a loop or heap.

A — Reverse Linked List (Full & Partial)

Full Reversal

Standard three-pointer iterative approach. Covered in detail in Part 3. Time: O(n), Space: O(1). New head is prev after the loop.

Partial Reversal (Reverse between positions left and right)

This is significantly harder. You must locate the node just before position left, reverse exactly (right - left + 1) nodes, then reconnect the reversed segment back into the list.

ListNode reverseBetween(ListNode head, int left, int right) {
    ListNode dummy = new ListNode(0);
    dummy.next = head;
    ListNode pre = dummy; // node just before the reversal zone

    for (int i = 0; i < left - 1; i++) pre = pre.next;

    ListNode curr = pre.next; // first node in the reversal zone
    ListNode prev = null;

    for (int i = 0; i < right - left + 1; i++) {
        ListNode next = curr.next;
        curr.next = prev;
        prev = curr;
        curr = next;
    }
    // Reconnect: pre.next is still the old first node of the segment
    pre.next.next = curr; // old first → node after reversal zone
    pre.next = prev;      // pre → new first (old last of segment)

    return dummy.next;
}
// Common mistakes: miscounting iterations, reconnecting in wrong order.
// pre.next.next = curr must come BEFORE pre.next = prev
// (otherwise pre.next is already overwritten to prev, losing the tail connection)

B — Fast & Slow Pointer Patterns

Cycle Detection (Floyd's Algorithm)

If fast and slow meet, there's a cycle. If fast reaches null, there's no cycle.

boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) return true; // met — cycle exists
    }
    return false; // fast reached null — no cycle
}

Find Cycle Start (LeetCode 142)

After detecting meeting point: reset slow to head, keep fast at meeting point. Move both one step at a time. They meet exactly at the cycle start. This is a mathematical property of Floyd's algorithm — take it as given in interviews.

ListNode detectCycleStart(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next; fast = fast.next.next;
        if (slow == fast) { // cycle detected
            slow = head;    // reset slow to head
            while (slow != fast) { // both move 1 step
                slow = slow.next; fast = fast.next;
            }
            return slow; // cycle start
        }
    }
    return null;
}

Find Middle of Linked List

When fast reaches end, slow is at the middle. For even-length lists, slow lands at the second middle node.

ListNode findMiddle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    return slow; // middle node
}
// For first middle node of even list: use fast.next != null as only condition.

C — Merge Two Sorted Lists

Two-pointer approach: maintain a pointer to the current tail of the merged list. At each step, attach whichever of the two current nodes is smaller.

ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0);
    ListNode curr = dummy;

    while (l1 != null && l2 != null) {
        if (l1.val <= l2.val) {
            curr.next = l1;   // attach smaller node
            l1 = l1.next;     // advance l1
        } else {
            curr.next = l2;
            l2 = l2.next;
        }
        curr = curr.next;     // advance merged tail
    }
    curr.next = (l1 != null) ? l1 : l2; // attach remaining list
    return dummy.next;
}
// Key insight: the remaining list is already sorted — just attach it.
// Common mistake: not handling when one list is exhausted before the other.

D — K-way Merge (Min-Heap)

Given K sorted linked lists, merge into one sorted list. Naively merging pairwise is O(nK log K). The optimal approach uses a min-heap (priority queue).

Approach: Insert the head of each list into a min-heap. Repeatedly extract the minimum node, attach it to the result, then insert that node's next into the heap.

ListNode mergeKLists(ListNode[] lists) {
    PriorityQueue<ListNode> pq = new PriorityQueue<>(
        (a, b) -> a.val - b.val // min-heap by node value
    );
    for (ListNode node : lists) {
        if (node != null) pq.offer(node); // seed with each list's head
    }
    ListNode dummy = new ListNode(0);
    ListNode curr = dummy;

    while (!pq.isEmpty()) {
        ListNode min = pq.poll();  // extract minimum
        curr.next = min;
        curr = curr.next;
        if (min.next != null) pq.offer(min.next); // push next into heap
    }
    return dummy.next;
}
// Time: O(n log K) where n = total nodes, K = number of lists
// Space: O(K) for the heap

E — Palindrome Linked List

Classic compound problem using three sub-techniques: find middle → reverse second half → compare both halves.

boolean isPalindrome(ListNode head) {
    // Step 1: Find middle
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next; fast = fast.next.next;
    }

    // Step 2: Reverse second half
    ListNode prev = null, curr = slow;
    while (curr != null) {
        ListNode next = curr.next;
        curr.next = prev; prev = curr; curr = next;
    }

    // Step 3: Compare first and reversed second half
    ListNode left = head, right = prev;
    while (right != null) { // right is shorter or equal length
        if (left.val != right.val) return false;
        left = left.next; right = right.next;
    }
    return true;
}
// This modifies the list. Restore if required by the problem.

F — Reorder List

Given 1→2→3→4→5, reorder to 1→5→2→4→3. Strategy: find middle → reverse second half → interleave the two halves.

void reorderList(ListNode head) {
    if (head == null || head.next == null) return;

    // 1. Find middle
    ListNode slow = head, fast = head;
    while (fast.next != null && fast.next.next != null) {
        slow = slow.next; fast = fast.next.next;
    }

    // 2. Reverse second half
    ListNode second = slow.next;
    slow.next = null; // sever first half from second
    ListNode prev = null;
    while (second != null) {
        ListNode next = second.next;
        second.next = prev; prev = second; second = next;
    }

    // 3. Interleave first half and reversed second half
    ListNode first = head;
    second = prev; // now head of reversed second half
    while (second != null) {
        ListNode tmp1 = first.next, tmp2 = second.next;
        first.next = second;
        second.next = tmp1;
        first = tmp1; second = tmp2;
    }
}

G — Intersection of Two Linked Lists

Two lists that share a tail (converge at a node). Find the intersection node.

Insight: If you traverse both lists and when you hit the end of one, continue from the head of the other — after at most two traversals, both pointers will have traveled the same total distance and will meet at the intersection node (or both reach null simultaneously if no intersection).

ListNode getIntersectionNode(ListNode headA, ListNode headB) {
    ListNode a = headA, b = headB;
    while (a != b) {           // continue until they meet (or both null)
        a = (a == null) ? headB : a.next; // switch to other list at end
        b = (b == null) ? headA : b.next;
    }
    return a; // intersection node, or null if no intersection
}
// Elegant — no length calculation needed.
// Both pointers travel: lenA + lenB total steps before meeting.

Reversal Template

// Iterative reversal — reusable in any problem
ListNode reverse(ListNode head) {
    ListNode prev = null;  // ① will become new tail (points to null)
    ListNode curr = head;  // ② current node being processed

    while (curr != null) {
        ListNode next = curr.next; // SAVE: don't lose the rest of list
        curr.next = prev;          // FLIP: point backward
        prev = curr;               // ADVANCE: prev moves forward
        curr = next;               // ADVANCE: curr uses saved reference
    }
    return prev; // ③ prev is new head; curr is null (past the end)
}
// Common mistakes:
// 1. Returning curr instead of prev
// 2. Forgetting to save next before overwriting curr.next
// 3. Starting prev = head instead of prev = null

Fast & Slow Pointer Template

// Core template — adapt for your specific problem
ListNode slow = head;
ListNode fast = head;

// Variant A: Find middle (fast reaches null when slow is at middle)
while (fast != null && fast.next != null) {
    slow = slow.next;
    fast = fast.next.next;
}
// slow is at middle

// Variant B: Cycle detection (check equality before advancing)
while (fast != null && fast.next != null) {
    slow = slow.next;
    fast = fast.next.next;
    if (slow == fast) { /* cycle found */ }
}

// Variant C: N steps ahead gap (for Nth from end)
for (int i = 0; i < n; i++) fast = fast.next;
while (fast != null) { slow = slow.next; fast = fast.next; }
// slow is now at Nth from end

// Critical: ALWAYS check both fast != null AND fast.next != null
// Checking only fast != null causes NPE when accessing fast.next.next

Merge Two Lists Template

// Reusable merge — always use a dummy node
ListNode merge(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0); // ① dummy: avoids head special case
    ListNode tail = dummy;          // ② tail: always points to end of merged list

    while (l1 != null && l2 != null) {
        if (l1.val <= l2.val) {    // ③ pick smaller
            tail.next = l1;
            l1 = l1.next;
        } else {
            tail.next = l2;
            l2 = l2.next;
        }
        tail = tail.next;            // ④ always advance tail
    }
    tail.next = (l1 != null) ? l1 : l2; // ⑤ attach remaining
    return dummy.next;               // ⑥ skip the dummy
}
// Common mistakes:
// 1. Forgetting to advance tail after each attachment
// 2. Forgetting to attach the remaining non-null list at the end
// 3. Using head instead of dummy — causes special-case logic for empty merge
EDGE CASES

Edge Cases & Pitfalls

Test every linked list solution against these cases before finalizing. Most bugs hide here.

!
Empty list (head == null). Almost every function needs an upfront null check. if (head == null) return null; or if (head == null) return;. Forgetting this causes NPE before you even start.
!
Single node list. Many patterns break here. Reversal of one node: should return the same node, not null. Fast & slow: fast is already at end before the loop starts. Palindrome: always true. Always dry run single-node input.
!
Two node list. Particularly tricky for reversal and palindrome. After reversal: head should be the old second node. Palindrome of [1,2]: false. [1,1]: true. Fast pointer starts at head, after one step fast.next is null — the loop may not even execute once in certain variants.
!
Cycle presence. If a cycle exists and your solution assumes linear traversal, it will loop infinitely. Always use Floyd's fast & slow when cycle presence is uncertain, or verify cycle-free before traversal if the problem guarantees it.
!
Updating head incorrectly after modification. After reversal, the new head is prev, not the original head. After deletion of the head node, you must update the head. Always use a dummy node or explicitly handle head updates when the head might change.
!
Nth from end where N equals list length. This deletes the head. With the fast & slow approach using a dummy node, this is handled correctly — slow remains at dummy, and slow.next = slow.next.next removes the head. Dry run this explicitly.
!
Merging two lists when one is empty. mergeTwoLists(null, l2) should return l2. The dummy node + while loop naturally handles this — the while loop body never executes, and tail.next = l2 attaches the entire second list.
!
Losing the tail reference after reversal. In palindrome check or reorder list, after reversing the second half, the original middle node's next still points into the reversed portion. If you don't sever the lists before reversing (slow.next = null), you may create a cycle unintentionally.
!
Even vs odd length in palindrome/middle problems. Odd length: one true middle. Even length: slow lands on the second of the two middles. Ensure your algorithm is verified against both [1,2,1] and [1,2,2,1].

Linked Lists vs Arrays

Dimension Array Linked List
Memory layout Contiguous — cache friendly Scattered — cache unfriendly
Access by index O(1) O(n)
Insert at head O(n) — shift all O(1)
Delete in middle O(n) — shift all O(1) given prev pointer
Memory per element Value only Value + 1–2 pointers
Dynamic size ArrayList resizes (amortized) True O(1) per insert/delete

Binary search O(log n) Not possible — no random access
Reversal O(n) — simple swap with indices O(n) — pointer rewiring, more error-prone

Two Pointers: Arrays vs Linked Lists

Two-pointer problems look similar between arrays and linked lists, but the mechanics differ significantly.

  • Arrays: Two pointers use indices. You can jump to arr[i] and arr[j] instantly. Can move backward. Partitioning and sorting leverage random access.
  • Linked Lists: Two pointers move by following next. No random access. Cannot move backward (singly). Fast & slow exploit the speed difference. You cannot "skip back" — plan ahead.
When Linked Lists Win
Frequent insertions/deletions in the middle of a large collection. LRU cache (O(1) delete of any node). Queues and deques built from scratch. When you never need random access. Any situation where shifting elements in an array would dominate runtime.
When Arrays Win
Any problem requiring index access. Binary search. Sorting (most algorithms work on arrays). Cache performance matters (tight loop over millions of elements). Two-pointer with backward movement. If in doubt, reach for the array — linked lists have constant-factor overhead.

Recursion vs Iteration in Linked Lists

Dimension Recursive Iterative
Reversal clarity Elegant but non-obvious Mechanical and clear
Space complexity O(n) call stack O(1)
Stack overflow risk Yes — large lists crash No
Forward processing Natural but uses O(n) stack Natural, O(1) space
Backward processing Natural — unwind stack after recursion Need explicit stack or reversal first
Interview preference Show if asked, or for elegance Preferred for production correctness
The One Case for Recursion in Linked Lists
When you need to process a linked list in reverse order without first reversing it. For example, printing values from tail to head, or comparing a list against itself from both ends simultaneously. Recursion unwinds from the tail naturally — an iterative approach would require an explicit stack or prior reversal.

Progressive Problem Set — 38 Problems

Problems organized from beginner to advanced. Each entry includes the pattern tag, a 1–2 line directional hint, and what makes it tricky. No full solutions — the struggle is the learning.

Beginner — 12 Problems
#1
206 — Reverse Linked List Easy Reversal · Iterative
Classic three-pointer iterative reversal. Start with prev=null, curr=head. Save next, flip link, advance both.
The #1 linked list problem. Must know cold. Returning curr (null) instead of prev is the single most common mistake. Try recursive version too.
#2
876 — Middle of the Linked List Easy Fast & Slow
Fast & slow pointers. When fast reaches the end, slow is at the middle.
For even-length lists, slow lands on the second middle. The problem specifies which one you need — read carefully. Subtle difference in while condition: fast.next != null vs fast != null.
#3
141 — Linked List Cycle Easy Fast & Slow · Cycle
Fast and slow. If they ever point to the same node, there's a cycle. If fast reaches null, no cycle.
Must check both fast != null AND fast.next != null before the step. Just one check causes NPE. Also — why does Floyd's algorithm work? Think about it. The answer builds intuition for LeetCode 142.
#4
21 — Merge Two Sorted Lists Easy Merge · Dummy Node
Use a dummy node and a tail pointer. At each step pick the smaller head node, attach it, advance the corresponding pointer.
Not attaching the remaining non-exhausted list at the end is the most common oversight. Try both iterative and recursive — the recursive solution is beautifully concise.
#5
83 — Remove Duplicates from Sorted List Easy Traversal · Pointer Skip
Traverse and skip nodes where curr.val == curr.next.val. No dummy node needed since the head is never removed.
Not advancing curr correctly when consecutive duplicates exist (e.g., 1→1→1). Must keep pointing curr.next until a different value appears.
#6
237 — Delete Node in a Linked List Medium Pointer Trick
You're not given the head — only the node to delete. Copy the next node's value into this node, then skip the next node.
Forces you to rethink what "deleting a node" means. The trick: node.val = node.next.val; node.next = node.next.next. Only works because the node is guaranteed not to be the tail.
#7
203 — Remove Linked List Elements Easy Traversal · Dummy Node
Use a dummy node before head. Traverse with prev pointer. When prev.next.val matches target, skip it.
Without a dummy node, you'd need to special-case removing the head. The dummy node eliminates that. Great intro problem for the dummy node pattern.
#8
92 — Reverse Linked List II Medium Partial Reversal
Find the node before position left. Run the standard reversal exactly (right-left+1) times. Reconnect: pre.next.next = curr, pre.next = prev.
The reconnection order matters: set pre.next.next BEFORE pre.next. If you reverse the assignment order, you lose the reference to curr. One of the trickier pointer problems.
#9
142 — Linked List Cycle II Medium Fast & Slow · Cycle Start
Detect cycle with fast & slow. Then reset slow to head, keep fast at meeting point. Move both by 1 step — they meet at cycle start.
The "reset slow to head" step feels magical without the math. Understanding why this works (distance properties of Floyd's) is a great interview talking point. The interviewer almost always asks why.
#10
19 — Remove Nth Node From End of List Medium Fast & Slow · Gap
Dummy node + fast & slow with a gap of N+1 (not N) steps. When fast is null, slow is at the node before the target.
The gap is N+1, not N — you need slow to be at the predecessor of the node to delete. Using N steps instead stops slow at the target itself. Also handles the case where N equals list length (removes head).
#11
160 — Intersection of Two Linked Lists Easy Two Pointer · Cycle Switch
Two pointers that switch to the other list's head when they hit null. They'll meet at the intersection after equal total traversal distance.
Why it works: both pointers travel lenA + lenB total steps. If there's an intersection, they meet at it. If not, both reach null simultaneously. The key insight: equal total distance, not equal step count.
#12
234 — Palindrome Linked List Easy Fast & Slow · Reversal · Compare
Find middle with fast & slow. Reverse second half in-place. Compare first half with reversed second half node by node.
A compound problem testing three techniques. Tricky parts: even vs odd length handling, severing the list at the midpoint before reversal, and iterating until right (reversed half) is null — not left.
Intermediate — 14 Problems
#13
143 — Reorder List Medium Find Middle · Reverse · Interleave
Three steps: find middle, reverse second half, interleave the two halves by alternating nodes.
The interleaving step requires careful pointer juggling (four pointers at once). Must sever the two halves before reversing (slow.next = null). The most coordination-heavy pattern on this list.
#14
82 — Remove Duplicates from Sorted List II Medium Dummy Node · Two Pointer Skip
Use a dummy node. Track prev. If curr.val == curr.next.val, skip ALL nodes with that value — not just one.
Must skip every node with a duplicate value, including the first occurrence (unlike LeetCode 83). Need a flag or inner while loop to consume all duplicates before reconnecting prev.
#15
2 — Add Two Numbers Medium Traversal · Carry Simulation
Traverse both lists simultaneously. At each step: sum = l1.val + l2.val + carry. New node value = sum % 10. carry = sum / 10.
Three edge cases: lists of different lengths, carry remaining after both lists are exhausted (must create one more node), and initializing carry=0 correctly.
#16
86 — Partition List Medium Two Dummy Lists · Split & Join
Create two dummy-headed lists: one for nodes less than x, one for nodes ≥ x. Traverse once, appending each node to the appropriate list. Join the two lists.
Must null-terminate the "greater" list before joining — otherwise the tail of the greater list may still point to something in the original list, creating a cycle.
#17
61 — Rotate List Medium Length + Modulo + Circular
Find list length, compute effective rotation k = k % length. Make it circular, find new tail at position (length - k), break the circle there.
k can be larger than the list length — must take modulo first. New head is at position (length - k + 1). Off-by-one in finding the new tail is very common.
#18
138 — Copy List with Random Pointer Medium HashMap · Clone
Two-pass with HashMap: first create all cloned nodes, second assign next and random pointers using the map.
The random pointer can point to any node including null or the node itself — handle null explicitly. Also solvable without extra space by interleaving cloned nodes, but that's harder to implement correctly.
#19
24 — Swap Nodes in Pairs Medium Dummy Node · Pair Swap
Use a dummy node. For each pair: save pointers, rewire the pair to swap them, advance by 2 nodes. Repeat.
Four pointers involved in each swap (prev, first, second, next). Updating them in the wrong order loses reference to the next pair. Good exercise in disciplined pointer sequencing.
#20
148 — Sort List Medium Merge Sort · Find Middle
Merge sort on linked lists: find middle, split, recursively sort each half, merge the two sorted halves.
Finding the middle correctly with fast & slow is critical — must use the "first middle" variant (while fast.next.next != null) to avoid infinite recursion. Merge sort is the natural O(n log n) sort for linked lists (no random access needed).
#21
328 — Odd Even Linked List Medium Two Pointer · Reorder
Maintain two separate pointers: one for odd-indexed nodes, one for even-indexed nodes. Weave them through the list, then join odd tail to even head.
Must save the even head before modifying pointers. Null checks for even and even.next are critical (odd-length lists). The indices are 1-based positions, not values.
#22
445 — Add Two Numbers II Medium Stack · Reverse Traversal
Use stacks to process from least significant digit. Pop from both stacks, compute sum + carry, build result in reverse.
Lists are not reversed — digits are in natural order (most significant first). Stacks give you access to the tail first. Alternatively, reverse both lists, add, then reverse result — but stacks are cleaner.

430 — Flatten a Multilevel Doubly Linked List Medium DFS · Doubly Linked
Iterative DFS: when a node has a child, insert the child's entire chain between the current node and its next. Use a stack or pointer-based approach.

#24
1721 — Swapping Nodes in a Linked List Medium Fast & Slow · Value Swap
Find Kth node from start with a counter. Find Kth from end using fast & slow gap. Swap their values (not nodes).
Swapping values is much simpler than swapping nodes. If you try to swap actual nodes, you deal with many pointer updates. Confirming K from end alignment via the gap technique is the main challenge.
#25
287 — Find the Duplicate Number Medium Floyd's Cycle · Array as List
Treat array values as next pointers (index → arr[index]). Duplicate creates a cycle. Apply Floyd's algorithm to find cycle start = duplicate.
Recognizing that an array can be modeled as a linked list with implicit pointers is a huge insight. The duplicate must be the cycle entry point. Uses exactly the same logic as LeetCode 142 — connects two seemingly unrelated problems.
#26
707 — Design Linked List Medium Implementation · All Operations
Implement a singly linked list with get, addAtHead, addAtTail, addAtIndex, deleteAtIndex. Use a dummy head node and maintain a size counter.

23 — Merge K Sorted Lists Hard K-way Merge · Min-Heap

Null-checking before adding to heap. The comparator must handle node comparison correctly. Divide-and-conquer (pairwise merge) is an alternative O(n log K) approach without a heap — know both.

The hardest pure linked list pointer problem. You need to: check K nodes exist, reverse exactly K, reconnect to the result of the next group. The recursive version is cleaner but O(n/K) stack space. Iterative needs careful tracking of group boundaries.

Groups have sizes 1, 2, 3, ... but the last group may be shorter. Reverse groups whose actual size is even. Track group boundaries carefully.
